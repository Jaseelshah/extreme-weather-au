# Australian Extreme Weather Pipeline

I built this over a couple of weekends because I wanted to get better at end-to-end data pipelines and also because Australian weather data is surprisingly hard to work with in bulk.

It scrapes storm records from BOM and financial catastrophe data from ICA, loads everything into SQLite, runs some pre-aggregations, and serves a Plotly Dash dashboard. Covers 2005–present — about 8,600 storm events and $50B+ in insured losses.

---

## What it does

- Pulls storm records from the [BOM Severe Storms Archive](http://www.bom.gov.au/australia/stormarchive/) — one request per hazard type (rain, hail, wind, etc.)
- Downloads the [ICA Historical Catastrophe List](https://insurancecouncil.com.au/resource/historical-catastrophe-list/) spreadsheet and parses the financial data
- Stores everything in a local SQLite database with idempotent inserts so re-runs never create duplicates
- Pre-computes summary tables so the dashboard doesn't have to do GROUP BY on every load
- Runs monthly on macOS via launchd

## Quick start

```bash
git clone https://github.com/Jaseelshah/extreme-weather-au.git
cd extreme-weather-au

python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# Run the full pipeline
python automation/run_pipeline.py

# Launch the dashboard
python dashboards/app.py
# → http://localhost:8050
```

If BOM or ICA are unreachable, the pipeline falls back to the snapshot files in `data/` automatically.

## Visualisation

### Plotly Dash (interactive, runs locally)

Three-page dark-themed dashboard at `http://localhost:8050`:

**Monthly Events** — stacked bar and area charts of storm frequency by state and hazard type. Has a state filter dropdown.

**Financial Impact** — annual insured losses by state, significant events table. Year range slider to zoom in.

**Hazard Map** — donut chart of event type breakdown, geo scatter map of event locations across Australia, monthly summary table.

### Power BI

For Power BI, run the CSV export script first:

```bash
python export_for_powerbi.py
```

This dumps all analytics tables and raw data into `data/exports/`:

- `monthly_events_by_state.csv` — monthly event totals per state per hazard type
- `monthly_events_by_hazard.csv` — monthly totals per hazard type (all states)
- `annual_financial_by_state.csv` — annual losses and claims per state (last 5 years)
- `significant_events.csv` — costliest event per state per year
- `events_by_hazard_type.csv` — all-time event counts per hazard type
- `storm_events_raw.csv` — full raw BOM data (8,600+ rows)
- `financial_impacts_raw.csv` — full raw ICA data

In Power BI: **Get Data → Text/CSV**, point it at these files. The pre-aggregated tables map directly to the required dashboard views (monthly totals per state, financial impact summary, month-by-month data table).

## Challenges & Learnings

**BOM form parameters took a while.** The BOM storm archive is a CGI form that returns CSV — but only if you hit it with exactly the right query parameters. The first few attempts returned HTML error pages. Had to inspect the form submission manually to figure out the right field names, then test each storm type separately because they have slightly different response formats for waterspouts and dust devils.

**ICA header row detection.** The ICA spreadsheet has a multi-row title block at the top — the actual column headers aren't on row 1, they're buried around row 10. And that row number shifts slightly between annual editions when they update the title section. Ended up writing a scanner that looks for the row containing "cat name" and uses that as the header, which has held up across the last three editions.

**Multi-state events in financial data.** ICA lists events like "NSW, QLD, VIC" as a single row. For the financial charts I needed per-state breakdowns, but ICA doesn't publish per-state loss figures. My workaround was to attribute the full insured loss to each affected state — it overstates totals if you add them up, but it's the only way to show geographic distribution without fabricating data. I added a note about this in the transformations file.

**SQLite WAL mode matters.** Initially the dashboard would occasionally get a "database is locked" error when it tried to read while the pipeline was mid-write. Enabling WAL journal mode fixed it — WAL allows concurrent reads during writes, which SQLite's default journal mode doesn't.

**Offline resilience was an afterthought.** I originally didn't have fallback logic and the first time BOM had a maintenance window the pipeline silently produced an empty database. Added the CSV/xlsx snapshot fallback after that.

## Data sources

| Source | What it covers | Format |
|--------|---------------|--------|
| BOM Severe Storms Archive | 1795–present, storm events by location | CSV via HTTP form |
| ICA Catastrophe List | 1967–present, insured losses by event | .xlsx download |

Pipeline filters both to 2005+ for consistency.

**Why these two sources?** I evaluated the three suggested sites plus a few others. DisasterAssist.gov.au and the NSW disaster declarations page are both useful for policy/recovery information but don't have structured, scrapeable data with lat/long coordinates and financial figures — they're more like news feeds. The BOM storm archive is the only source I found with per-event geographic coordinates across all of Australia, and ICA is the only public source with per-event insured loss figures. Together they cover the storm event detail (BOM) and the financial impact (ICA) that the case study asks for.

**A note on impact/injury data:** BOM doesn't publish injury or fatality counts — those are tracked by state emergency services and aren't part of the storm archive. For impact metrics I'm relying on ICA data which gives insurance claim counts and total insured losses per event. If I were to extend this, the Emergency Management Australia disaster database would be the next source to add for injury/fatality numbers.

## Tests

```bash
python -m pytest tests/ -v
```

37 tests covering the parsing logic that's most likely to break: state string splitting, year extraction from messy ICA cells, BOM CSV parsing edge cases (empty input, HTML error pages), safe type conversion, and a deduplication integration test that verifies INSERT OR IGNORE on an in-memory database.
